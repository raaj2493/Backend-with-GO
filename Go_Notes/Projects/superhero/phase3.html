🤔 Can Go Speak PostgreSQL?

No. Go only knows database/sql — a generic database API. Think of it like a Universal Remote Control — it knows "Turn On" and "Volume Up" but does not know how to talk to a specific TV.

🔌 Why Drivers Exist ⚡

When database/sql says db.Query(...), who converts that into PostgreSQL protocol packets?

Answer: The Driver! It sits between database/sql and PostgreSQL, translating Go function calls into database-specific language.

Go Code
→
database/sql
→
Driver
→
PostgreSQL

⚡ Driver We Will Use: pgx

We will use github.com/jackc/pgx/v5 — Modern, Fast, Widely used, Production-ready.

What Is sql.DB? (Big Misconception!)

❌ Many beginners think: db *sql.DB means "One Database Connection"

✅ The truth: db *sql.DB is actually a Connection Pool Manager!

Internal Architecture:

sql.DB
 ├── Connection #1
 ├── Connection #2
 ├── Connection #3
 ├── Connection #4
 └── Connection #5

It manages many connections. Not one.

❌ Without Pooling

  • 100 requests arrive
  • Open Connection → Run Query → Close Connection
  • 100 times!
  • Slow. Very slow.

✅ With Pooling

  • 100 requests arrive
  • Reuse Existing Connections
  • No opening/closing overhead
  • Fast. Efficient. Professional.

Create Config Package

Why? Never hardcode localhost, 5432, postgres inside code. Configuration should live separately.

Create: internal/config/config.go

package config

import "os"

type Config struct {
    DBHost     string
    DBPort     string
    DBUser     string
    DBPassword string
    DBName     string
}

func Load() Config {
    return Config{
        DBHost:     getEnv("DB_HOST", "localhost"),
        DBPort:     getEnv("DB_PORT", "5432"),
        DBUser:     getEnv("DB_USER", "postgres"),
        DBPassword: getEnv("DB_PASSWORD", "postgres"),
        DBName:     getEnv("DB_NAME", "superhero_db"),
    }
}

func getEnv(key string, fallback string) string {
    value := os.Getenv(key)
    if value == "" {
        return fallback
    }
    return value
}

💡 Why getEnv exists: If DB_HOST does not exist, instead of crashing, the fallback is used. Safe and reliable!

5

Create Database Package

This is the most important file of the phase! It creates the connection between Go and PostgreSQL.

Create: internal/database/postgres.go

package database

import (
    "database/sql"
    "fmt"
    _ "github.com/jackc/pgx/v5/stdlib"
    "superhero-api/internal/config"
)

func NewPostgresDB(cfg config.Config) (*sql.DB, error) {
    dsn := fmt.Sprintf(
        "host=%s port=%s user=%s password=%s dbname=%s sslmode=disable",
        cfg.DBHost, cfg.DBPort, cfg.DBUser, cfg.DBPassword, cfg.DBName,
    )
    db, err := sql.Open("pgx", dsn)
    if err != nil {
        return nil, err
    }
    err = db.Ping()
    if err != nil {
        return nil, err
    }
    return db, nil
}

⚠️ Deep Dive: sql.Open()

Most beginners think sql.Open() creates a database connection. Not exactly!

What sql.Open() Really Does

1️⃣Creates a Connection Pool Manager (sql.DB)
2️⃣Stores the Driver (pgx)
3️⃣Stores Connection Info (DSN string)

Important: No actual connection is made yet! Just the manager is created.

🔍 Why Ping Exists

err = db.Ping()

db.Ping() forces Go to verify: "Can I actually reach PostgreSQL?"

Without Ping: sql.Open() may succeed even if PostgreSQL is down! It only validates the DSN string format, not the actual connection.

Repository Pattern

❌ Bad Approach

Handler → Raw SQL everywhere. Messy and unmaintainable!

✅ Good Approach

Handler → Repository → Database. SQL stays in one place!

Create: internal/repository/superhero_repository.go

package repository

import (
    "database/sql"
    "superhero-api/internal/models"
)

type SuperheroRepository struct {
    DB *sql.DB
}

func NewSuperheroRepository(db *sql.DB) *SuperheroRepository {
    return &SuperheroRepository{DB: db}
}

func (r *SuperheroRepository) GetAll() ([]models.Superhero, error) {
    rows, err := r.DB.Query(`SELECT id, name, real_name, universe, power_level, created_at FROM superheroes`)
    if err != nil {
        return nil, err
    }
    defer rows.Close()
    var superheroes []models.Superhero
    for rows.Next() {
        var hero models.Superhero
        err := rows.Scan(&hero.ID, &hero.Name, &hero.RealName, &hero.Universe, &hero.PowerLevel, &hero.CreatedAt)
        if err != nil {
            return nil, err
        }
        superheroes = append(superheroes, hero)
    }
    return superheroes, nil
}

🔍 Deep Dive: Query()

rows, err := db.Query(...)

Used when multiple rows are expected. Example: SELECT * FROM superheroes

Returns *sql.Rows — think of it as a cursor over results. You iterate through it row by row.

🔍 Deep Dive: rows.Next()

Moves the current row to the next row. Each call advances through the results.

Row 1
Iron Man
→
Row 2
Batman
→
Row 3
Thor

🔍 Deep Dive: Scan() — This Is Extremely Important!

Database row: 1 | Iron Man | Tony Stark | Marvel | 95

Scan copies column values into Go variables:

rows.Scan(&hero.ID, &hero.Name, &hero.RealName, &hero.Universe, &hero.PowerLevel, &hero.CreatedAt)

This process is called Struct Mapping — converting database rows into Go structs!

Create Main File

Create: cmd/server/main.go

package main

import (
    "fmt"
    "log"
    "superhero-api/internal/config"
    "superhero-api/internal/database"
    "superhero-api/internal/repository"
)

func main() {
    cfg := config.Load()
    db, err := database.NewPostgresDB(cfg)
    if err != nil {
        log.Fatal(err)
    }
    repo := repository.NewSuperheroRepository(db)
    heroes, err := repo.GetAll()
    if err != nil {
        log.Fatal(err)
    }
    fmt.Println("Superheroes:")
    for _, hero := range heroes {
        fmt.Println(hero)
    }
}
