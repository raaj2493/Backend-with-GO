🧩 What Is Each Layer?

Think of it like a restaurant kitchen — everyone has a job!

👤

Client

Examples: Postman, Browser, Frontend App, Mobile App

The client sends HTTP requests. Example: GET /superheroes

🌐

Gin Router

Receives incoming requests and decides where they go.

Example: GET /superheroes → Router decides → GetAllSuperheroesHandler

⚙️

Handler Layer

Handlers understand HTTP Requests, JSON, Status Codes, and URL Parameters.

Example: GET /superheroes/5 → Handler extracts id = 5 → Asks Repository: "Please fetch superhero 5"

📂

Repository Layer

Contains all database logic and SQL queries. 

SELECT * FROM superheroes
WHERE id = 5;

Key Rule: The handler should NOT know SQL. The repository should. This separation is called Separation of Concerns.

🔗

database/sql

Go's standard database package. Provides db.Query(), db.QueryRow(), db.Exec().

It does NOT understand PostgreSQL itself. It needs a driver to speak PostgreSQL.

🐘

PostgreSQL

Stores data permanently. This is our source of truth.

🐳 Docker Strategy

Instead of installing PostgreSQL directly on your machine, we'll run it inside Docker. One command: docker compose up — everything starts!

❌ Without Docker

  • Install PostgreSQL manually
  • Configure PostgreSQL
  • Manage versions
  • Fix environment issues
  • Lots of pain 😫

✅ With Docker

  • docker compose up
  • Everything starts automatically
  • Same environment everywhere
  • Easy to clean up
  • Simple! 😊

📦 Key Docker Concepts 🐳

1
Image

A blueprint. Example: postgres:17. Like a recipe — not running yet.

2
Container

A running image. Like: Image = Class, Container = Object

3
Volume

Persistent storage. Without it: container deleted = database deleted. Bad!

4
Port Mapping

PostgreSQL inside container runs on 5432. We expose it as localhost:5432 so Go can connect.

5
Docker Compose

Instead of long docker run ... commands, we create a docker-compose.yml file and run docker compose up.

🗄️ Database Design

We need one table: superheroes

Table: superheroes

Column Type Why?
id SERIAL Unique identifier, auto-increments (1, 2, 3...)
name VARCHAR Hero name: "Iron Man", "Batman"
real_name VARCHAR Secret identity: "Tony Stark", "Bruce Wayne"
universe VARCHAR "Marvel" or "DC"
power_level INTEGER Simple numeric ranking: 95, 80, 70
created_at TIMESTAMP When the hero was added to the database

💡 Tip: id is our Primary Key — it uniquely identifies each row. No two heroes can have the same ID!

🐹 Go Strategy

We'll build this API using Go's best tools and patterns. Here's our tech stack:

1
Gin

For the HTTP API and routing. Fast and beginner-friendly!

2
database/sql

Go's standard package for database communication.

3
PostgreSQL Driver

Allows Go to speak PostgreSQL (like a translator).

4
Environment Variables

For configuration (DB host, port, password) — no hardcoded secrets!

5
Repository Pattern

For clean architecture — keeping database logic separate from HTTP logic.

📁 Project Structure Roadmap

A well-organized folder structure makes your code easy to navigate and maintain!

Final Folder Tree 🌳

superhero-api/
│
├── cmd/
│   └── server/
│       └── main.go              // Application entrypoint
│
├── internal/
│   ├── config/
│   │   └── config.go            // Reads env variables
│   ├── database/
│   │   └── postgres.go          // DB connection
│   ├── models/
│   │   └── superhero.go         // Structs
│   ├── repository/
│   │   └── superhero_repository.go  // SQL lives here
│   ├── handlers/
│   │   └── superhero_handler.go // HTTP logic
│   ├── routes/
│   │   └── routes.go            // API endpoints
│   └── validation/
│       └── validator.go         // Input validation
│
├── migrations/                  // Database schema files
├── .env                           // Environment variables
├── Dockerfile                     // Go app container
├── docker-compose.yml             // Multi-container setup
├── go.mod                         // Go dependencies
└── README.md                    // Documentation

🧠 Remember: cmd/server/main.go is where func main() lives. Everything else is in internal/ — Go's convention for private application code.

Why This Structure? 🤔

📂
cmd/ — Application entrypoint. Contains func main()
⚙️
config/ — Reads environment variables like DB_HOST=localhost
🔗
database/ — Creates the PostgreSQL connection pool
📦
models/ — Contains structs like type Superhero struct {}
🗄️
repository/ — SQL lives here: INSERT, SELECT, UPDATE, DELETE
🌐
handlers/ — HTTP logic: POST /superheroes, GET /superheroes
🛣️
routes/ — Registers all API endpoints with Gin
📜
migrations/ — Database schema files like CREATE TABLE superheroes
