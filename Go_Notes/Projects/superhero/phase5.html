Problem #1: No Validation

Right now, this bad data gets inserted into the database:

{"name": "", "real_name": "", "universe": "", "power_level": 0}

Bad data - Bad database - Bad API

1

Install Validator

Install: go get github.com/go-playground/validator/v10

This library lets us add validation tags to our struct fields - like rules that automatically check if data is good!

1

Modify Model with Validation Tags

Validation tags are like little rule stickers on struct fields!

package models

import "time"

type Superhero struct {
    ID         int       `json:"id"`
    Name       string    `json:"name"`
    RealName   string    `json:"real_name"`
    Universe   string    `json:"universe"`
    PowerLevel int       `json:"power_level"`
    CreatedAt  time.Time  `json:"created_at"`
}

// Validation tags added!
type CreateSuperheroRequest struct {
    Name       string `json:"name" validate:"required"`
    RealName   string `json:"real_name" validate:"required"`
    Universe   string `json:"universe" validate:"required"`
    PowerLevel int    `json:"power_level" validate:"gte=1,lte=100"`
}

type UpdateSuperheroRequest struct {
    Name       string `json:"name" validate:"required"`
    RealName   string `json:"real_name" validate:"required"`
    Universe   string `json:"universe" validate:"required"`
    PowerLevel int    `json:"power_level" validate:"gte=1,lte=100"`
}

required

Field cannot be empty

gte=1

Greater Than or Equal to 1

lte=100

Less Than or Equal to 100

2

Create Validation Package

Why create once? Instead of validator.New() in every handler, we create it once and reuse everywhere!

// internal/validation/validator.go

package validation

import "github.com/go-playground/validator/v10"

// Global validator - create once, use everywhere!
var Validate = validator.New()

Think of it like a shared stamp - you do not make a new stamp every time you mail a letter!

3

Update Handler with Validation

Add validation right after ShouldBindJSON() - check the data BEFORE sending it to the database!

func (h *SuperheroHandler) Create(c *gin.Context) {
    var req models.CreateSuperheroRequest

    // Step 1: Parse JSON into struct
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }

    // Step 2: VALIDATE the struct!
    err := validation.Validate.Struct(req)
    if err != nil {
        c.JSON(http.StatusBadRequest,
            gin.H{"error": err.Error()})
        return
    }

    // Step 3: Only valid data reaches the database!
    if err := h.Repo.Create(req); err != nil {
        c.JSON(http.StatusInternalServerError,
            gin.H{"error": err.Error()})
        return
    }

    c.JSON(http.StatusCreated,
        gin.H{"message": "superhero created"})
}

Before (No Validation)

{"name":""} -> Database accepts it

After (With Validation)

{"error":"name is required"}

Do the same for Update() - validation protects ALL entry points!

5

Modify Config with .env Support

package config

import (
    "log"
    "os"
    "github.com/joho/godotenv"
)

type Config struct {
    DBHost     string
    DBPort     string
    DBUser     string
    DBPassword string
    DBName     string
}

func Load() Config {
    // Auto-load .env file!
    err := godotenv.Load()
    if err != nil {
        log.Println(".env file not found")
    }

    return Config{
        DBHost:     getEnv("DB_HOST", "localhost"),
        DBPort:     getEnv("DB_PORT", "5432"),
        DBUser:     getEnv("DB_USER", "postgres"),
        DBPassword: getEnv("DB_PASSWORD", "postgres"),
        DBName:     getEnv("DB_NAME", "superhero_db"),
    }
}

// Helper: get env var or use fallback
func getEnv(key string, fallback string) string {
    value := os.Getenv(key)
    if value == "" {
        return fallback
    }
    return value
}

Problem #3: Database Schema Management

Currently: CREATE TABLE was typed manually. Not scalable for teams!

6

What Are Migrations?

Think of migrations as "Git for your database schema" - version-controlled SQL changes!

Without Migrations

  • New dev joins -> "Ask Raj for SQL commands"
  • Schema changes are manual and error-prone
  • No history of changes
  • Production deploys are scary

With Migrations

  • New dev joins -> run migrations
  • Schema changes are versioned files
  • Full history in Git
  • Production deploys are smooth
7

Create Migration File

-- migrations/001_create_superheroes_table.sql

CREATE TABLE IF NOT EXISTS superheroes (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    real_name VARCHAR(100) NOT NULL,
    universe VARCHAR(50) NOT NULL,
    power_level INTEGER NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

New developer joins? Just run the migration tool - database created automatically!

Industry tools: golang-migrate, goose, atlas - for now, focus on understanding the concept!

Better Error Handling

Currently: "hero not found" and "database offline" both return 500. Not ideal!

sql.ErrNoRows - The Special Error

When SELECT * FROM superheroes WHERE id=999 returns nothing, Go returns sql.ErrNoRows. Use it to return 404 Not Found instead of 500!

Connection Pool Tuning

Remember: sql.DB is a pool manager - we can tune it!

// Tuning the connection pool
db.SetMaxOpenConns(25)        // Max 25 simultaneous connections
db.SetMaxIdleConns(5)          // Keep 5 ready for reuse
db.SetConnMaxLifetime(time.Hour) // Refresh connections every hour

5 Most Important Concepts

If you remember only five things, remember these:

1

sql.DB Is NOT A Connection

It is a Connection Pool Manager

2

Drivers Translate Between Go And PostgreSQL

Go -> database/sql -> pgx Driver -> PostgreSQL

3

Query(), QueryRow(), Exec()

Query() = Multiple rows. QueryRow() = Single row. Exec() = No returned rows.

4

Scan() Maps Columns To Go Variables

Database Row -> Scan() -> Struct

5

Repository Pattern Keeps SQL In One Place

Handler -> Repository -> Database (instead of SQL scattered everywhere)
