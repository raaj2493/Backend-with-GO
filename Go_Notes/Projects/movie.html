🧺 What Is a Slice? (Our Fake Database)

Think of a slice like a shopping basket 🛒

You can put things in, take things out, look at things inside.

In Go, a slice is a list that can grow and shrink.

// This is our "database" - a slice of movies
var movies = []Movie{
    {ID: "1", Title: "Interstellar", Director: "Christopher Nolan", Year: 2014},
    {ID: "2", Title: "Inception",    Director: "Christopher Nolan", Year: 2010},
}

⚠️ Important! This slice lives only in computer memory (RAM).

If you stop the server → all data is GONE. That's why real apps use databases like PostgreSQL.

But for learning? Slices are perfect! 👍

🧪 The Complete Code

Full CRUD Movies API in Go + Gin:

package main

import (
    "net/http"
    "github.com/gin-gonic/gin"
)

type Movie struct {
    ID       string `json:"id"`
    Title    string `json:"title"`
    Director string `json:"director"`
    Year     int    `json:"year"`
}

var movies = []Movie{
    {ID: "1", Title: "Interstellar", Director: "Christopher Nolan", Year: 2014},
    {ID: "2", Title: "Inception",    Director: "Christopher Nolan", Year: 2010},
}

func main() {
    router := gin.Default()

    router.GET("/movies",     getMovies)
    router.GET("/movies/:id", getMovieByID)
    router.POST("/movies",    createMovie)
    router.PUT("/movies/:id",  updateMovie)
    router.DELETE("/movies/:id", deleteMovie)

    router.Run(":8080")
}

This is your entire server! 🚀

When you run: go run main.go

Your API is live on localhost:8080 😄

Function 1: getMovies()

👆 This handles: GET /movies

🌍 Real Life Example:

You go to a library and say: "Show me ALL books you have."

Librarian opens the shelf and shows you every single book.

That is getMovies. It shows all movies from the slice.

func getMovies(c *gin.Context) {
    c.JSON(http.StatusOK, movies)
}

🔬 Let's dissect EVERY word:

Word 1: func

This is Go's keyword to say: "Hey, I am creating a function (a set of instructions)."

Just like a recipe has a name. func is how you write the recipe name in Go.

func bakeACake() {  // this is a function
    // put steps here
}

Word 2: getMovies

This is the name of our function. We chose this name to clearly say what this function does: it gets movies.

You could name it banana and it would still work. But good names = good code 😄

Word 3: (c *gin.Context)

This is the input our function gets. Gin automatically gives us this c (our magic backpack).

The * means it is a pointer — Gin gives us the real backpack, not a copy.

Word 4: c.JSON(http.StatusOK, movies)

This line does 2 things at once:

Part A: http.StatusOK

= the number 200

Means: "Everything went well!"

Like a thumbs up 👍 to the client

Part B: movies

This is our slice with all movies.

Gin converts it to JSON and sends it.

Like packing food into a box 📦

📬 What does the client receive?

[
  { "id": "1", "title": "Interstellar", "director": "Christopher Nolan", "year": 2014 },
  { "id": "2", "title": "Inception",    "director": "Christopher Nolan", "year": 2010 }
]

A JSON array (list) of all movies. Square brackets [ ] mean it is a list.

READ ONE 🔍

Function 2: getMovieByID()

👆 This handles: GET /movies/:id

🌍 Real Life Example:

You go to a library and say: "I want ONLY the book with ID number 2."

Librarian searches the shelf, finds book #2, and gives only that one to you.

func getMovieByID(c *gin.Context) {
    id := c.Param("id")          // Step 1: get the id from URL

    for _, movie := range movies {  // Step 2: loop through all movies
        if movie.ID == id {           // Step 3: if this movie's ID matches
            c.JSON(http.StatusOK, movie) // Step 4: send that movie
            return                      // Step 5: stop the function
        }
    }

    // Step 6: if loop finished and nothing found:
    c.JSON(http.StatusNotFound, gin.H{"message": "movie not found"})
}

🔬 Every Step Explained:

Step 1: id := c.Param("id")

The URL is: /movies/2

The :id part in the route /movies/:id is a placeholder. Like a blank box: /movies/___

Whatever the user puts there (like 2), Gin captures it.

c.Param("id") pulls out that value. So now id = "2" (a string).

URL:

/movies/2

Result:

id = "2"

Step 2: for _, movie := range movies

This is a for loop. It is like telling a robot:

"Go through EVERY box in the basket, one by one. Look at each movie."

Lap 1

Look at Interstellar (ID=1)

Lap 2

Look at Inception (ID=2)

Lap 3

No more movies → stop

What is _ (underscore)?

In Go, range gives you two things: index (position number) and value (the movie).

We don't need the position number here, so we use _ to say: "I don't care about this, throw it away."

Step 3: if movie.ID == id

We check: "Does this movie's ID match what the user asked for?"

Example: user wants "2". We check each movie:

❌Interstellar: movie.ID = "1" == "2"? → NO
✅Inception: movie.ID = "2" == "2"? → YES! Found it!

Step 4 + 5: c.JSON(http.StatusOK, movie) then return

We send that one movie as JSON response with status 200 (all good).

Then return = "Stop the function right now."

Without return, the code would keep running and also send the "not found" error below. That would be wrong!

❌ Without return — BAD

Sends movie → then ALSO sends "not found" → client gets confused

✅ With return — GOOD

Sends movie → STOPS → done. Perfect.

Step 6: c.JSON(http.StatusNotFound, gin.H{...})

If the loop finished and NO movie matched, we come here.

http.StatusNotFound = 404. You've seen 404 before! "Page not found." Same idea.

gin.H{"message": "movie not found"} sends a friendly message so the client knows what went wrong.

// Client receives:
{ "message": "movie not found" }   // with status 404
CREATE ✍️

Function 3: createMovie()

👆 This handles: POST /movies

🌍 Real Life Example:

You go to the library and say: "I have a NEW book to donate! Please add it to your collection."

You hand them the book. They add it to the shelf. Now the library has one more book.

func createMovie(c *gin.Context) {
    var newMovie Movie                       // Step 1: create empty container

    err := c.ShouldBindJSON(&newMovie)      // Step 2: fill container from request body
    if err != nil {                          // Step 3: if JSON was bad
        c.JSON(http.StatusBadRequest, gin.H{
            "error": err.Error(),
        })
        return
    }

    movies = append(movies, newMovie)       // Step 4: add to our slice
    c.JSON(http.StatusCreated, newMovie)   // Step 5: send back what was created
}

🔬 Every Step Explained:

Step 1: var newMovie Movie

We create an empty Movie container. Like getting an empty form from the post office:

newMovie = Movie{
    ID:       "",      // blank
    Title:    "",      // blank
    Director: "",      // blank
    Year:     0,       // blank (zero for numbers)
}

In the next step, we will fill this form with the data from the request.

Step 2: err := c.ShouldBindJSON(&newMovie)

This does the magic: reads the JSON body from the HTTP request and fills our empty Movie struct.

JSON body → Go struct 🔄

Client sends this JSON:

{
  "id": "3",
  "title": "Batman",
  "director": "Nolan",
  "year": 2008
}

After ShouldBindJSON:

newMovie = Movie{
  ID:       "3",
  Title:    "Batman",
  Director: "Nolan",
  Year:     2008,
}

Why &newMovie and not just newMovie?

& means: "Give Gin the real address of newMovie in memory."

Without &, Gin fills a copy and throws it away. Our variable stays empty. Bad!

With &, Gin fills the real one. We can use it. Good!

Step 3: if err != nil

nil in Go means: "nothing" / "empty" / "no error".

So err != nil means: "There IS an error."

When can an error happen? When the client sends bad JSON:

❌ BAD JSON (client mistake):

{ title: Batman }

Missing quotes around key

✅ GOOD JSON:

{ "title": "Batman" }

Proper JSON format

If error: we send status 400 Bad Request with an error message and return (stop the function).

Step 4: movies = append(movies, newMovie)

This is the most important line. This actually saves the movie!

append is a built-in Go function. It takes a slice and adds something to the end.

Before append vs After append:

Before 🛒

Interstellar
Inception

After ✅

Interstellar
Inception
Batman ← NEW!

The key part: movies = append(...) — we must reassign back to movies. Otherwise Go throws the result away!

Step 5: c.JSON(http.StatusCreated, newMovie)

http.StatusCreated = 201. This is slightly different from 200!

200 OK

Something already existed and you got it

201 Created

Something NEW was created because of your request

We send back the newMovie so the client can confirm exactly what was created (including its ID).

UPDATE ✏️

Function 4: updateMovie()

👆 This handles: PUT /movies/:id

🌍 Real Life Example:

You realize the book's release year is wrong. You go to the library and say:

"Please UPDATE the book with ID 2. Here is the corrected information."

Librarian finds book #2, erases the old info, writes the new info. Same book, new details.

func updateMovie(c *gin.Context) {
    id := c.Param("id")                 // Step 1: get id from URL

    var updatedMovie Movie                // Step 2: empty container for new data
    err := c.ShouldBindJSON(&updatedMovie) // Step 3: fill it from request body
    if err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }

    for i, movie := range movies {        // Step 4: loop, this time we need index!
        if movie.ID == id {               // Step 5: found the movie
            movies[i] = updatedMovie     // Step 6: replace it
            c.JSON(http.StatusOK, updatedMovie)
            return
        }
    }

    c.JSON(http.StatusNotFound, gin.H{"message": "movie not found"})
}

🔬 The New Things In This Function:

New thing: for i, movie := range movies

In getMovieByID we used _ to ignore the index. Here we use i because we NEED the position to replace the movie!

What is i?

It's the position number in the slice. Like the seat number in a cinema.

i=0

Interstellar

i=1

Inception

i=2

Batman (if created)

Key Line: movies[i] = updatedMovie

This is direct replacement. We go to position i in the slice and swap out the old movie for the new one.

Like replacing a book on a specific shelf position 📚

Before Update

movies[1] = Inception (year: 2010)

After Update

movies[1] = Inception (year: 2023) ✨

⚠️ Important difference!

In getMovieByID, we did: for _, movie := range movies

But movie there is a copy. Changing it does nothing to the slice.

That's why here we use movies[i] = updatedMovie — it modifies the actual slice directly.

DELETE 🗑️

Function 5: deleteMovie()

👆 This handles: DELETE /movies/:id

🌍 Real Life Example:

You go to the library and say: "Please REMOVE the book with ID 1. We don't want it anymore."

Librarian finds the book, pulls it off the shelf, and throws it away. It's gone forever.

The remaining books shift together to fill the gap.

func deleteMovie(c *gin.Context) {
    id := c.Param("id")              // Step 1: get id from URL

    for i, movie := range movies {    // Step 2: loop to find the movie
        if movie.ID == id {            // Step 3: found it!
            movies = append(          // Step 4: rebuild slice without it
                movies[:i],
                movies[i+1:]...
            )
            c.JSON(http.StatusOK, gin.H{
                "message": "movie deleted",
            })
            return
        }
    }

    c.JSON(http.StatusNotFound, gin.H{"message": "movie not found"})
}

🔬 The Tricky Delete Trick Explained:

The Magic Line: movies = append(movies[:i], movies[i+1:]...)

This is the classic Go delete trick. It looks scary but it's simple!

🍕 Pizza Slice Analogy

Imagine you have 3 pizza slices on a plate (positions 0, 1, 2). You want to remove slice at position 1.

You take all slices BEFORE position 1, and all slices AFTER position 1, then combine them:

i=0

🍕 Interstellar

i=1 ← DELETE THIS

🍕 Inception

i=2

🍕 Batman

movies[:i]

= everything before position i

🍕 [Interstellar]

positions 0 to i-1

+

movies[i+1:]

= everything after position i

🍕 [Batman]

positions i+1 to end

Result: movies = [Interstellar, Batman] ✅

Inception is gone! The two pieces are joined back together.

What is ... (the three dots)?

The ... is called spread operator (or unpacking).

Imagine you have a bag of apples and you want to put them in a fruit basket. You don't put the whole bag in. You empty the bag and put each apple separately.

Without ...

You put the BAG inside another bag. 🛍️ inside 🛍️ = nested mess

With ...

You unpack the items and add them one by one. Correct! ✅

In code: append(movies[:i], movies[i+1:]...) — the ... says "unpack movies[i+1:] as individual items, not as a slice".

Why no 204 No Content status?

Some developers send 204 on delete (means: "done, no body"). We used 200 OK with a message.

Both are correct! Using 200 + message is friendlier for the client — they know what happened.

// Client receives:
{ "message": "movie deleted" }   // status 200

🚦 HTTP Status Codes Cheatsheet

Every response has a status code. Like a report card grade for the request.

200

OK

Request worked. Here is your data.

201

Created

New thing was made. Here it is.

400

Bad Request

YOU (client) sent something wrong. Bad JSON, missing fields, etc.

404

Not Found

We looked for it but it doesn't exist. "Page not found" vibes.

500

Internal Server Error

SERVER messed up. Not your fault. The backend code crashed.

Easy rule to remember:

🟢 2xx = Happy! Everything worked.

🟡 4xx = Client did something wrong.

🔴 5xx = Server did something wrong.

📦 gin.H

gin.H is shorthand for map[string]interface{}

Use it when you want to send a quick custom JSON response without creating a full struct.

gin.H{
    "name": "Ashutosh",
    "age": 22,
}
// becomes: {"name": "Ashutosh", "age": 22}

When to use gin.H vs a struct?

Use gin.H for:

Simple one-off messages like errors, success confirmations

Example: gin.H{"message": "deleted"}

Use a struct for:

When you have a well-defined data shape (like Movie)

Example: c.JSON(200, movie)

🔗 ShouldBindJSON()

Most important function for POST and PUT.

c.ShouldBindJSON(&movie)

Reads JSON → fills struct automatically. ✨

📝 Before Binding

movie := Movie{}
// contains: {ID: "", Title: ""}
Empty struct ⬆️

✅ After Binding

movie := Movie{
    ID: "1",
    Title: "Interstellar",
}
Filled with data ⬆️

🎯 Why ShouldBindJSON Needs Pointer?

❌ Without Pointer

c.ShouldBindJSON(movie)

Gin fills a copy. Original is unchanged.

✅ With Pointer

c.ShouldBindJSON(&movie)

Gin fills the real thing. Works!

⚔️ BindJSON vs ShouldBindJSON

BindJSON

  • ⚠️ Auto-writes error response
  • ⚠️ Less control

ShouldBindJSON

  • ✅ Returns error, you handle it
  • ✅ More professional
  • ✅ Preferred in real projects

🏷️ JSON Tags

type Movie struct {
    ID string `json:"id"`
}

❌ Without Tag

{ "ID": "1" }

✅ With Tag

{ "id": "1" }

Why does this matter?

Frontend developers expect "id" (lowercase), not "ID" (uppercase). JSON tags bridge this gap.

🤔 Why Capital Letters in Struct?

ID string   // ✅ works - EXPORTED (public)
id string   // ❌ doesn't - unexported (private)

In Go, capital first letter = public (exported).

Gin uses encoding/json which can only see exported (public) fields.

Private fields are invisible to JSON. They will be missing in the output!

✅ Public (exported)

Seen by everyone, including JSON encoder

ID, Title, Year

❌ Private (unexported)

Hidden from other packages, JSON encoder can't see it

id, title, year

Route Param

/movies/10

Part of the URL path itself.

Used for: identifying a specific resource (which movie?)

Query Param

After the ? in URL.

Used for: filtering, sorting, optional extras

🚨 Common Mistakes (Don't Do These!)

❌ Mistake 1: Forgetting return after sending response

if movie.ID == id {
    c.JSON(200, movie)
    // forgot return! function continues...
}
c.JSON(404, ...) // ALSO runs! Two responses = error

❌ Mistake 2: Forgetting & in ShouldBindJSON

c.ShouldBindJSON(movie)   // no & = fills a copy = movie stays empty!
c.ShouldBindJSON(&movie)  // correct ✅

❌ Mistake 3: Not reassigning after append

append(movies, newMovie)         // result thrown away!
movies = append(movies, newMovie) // correct ✅

❌ Mistake 4: Lowercase struct fields

type Movie struct {
    id    string  // ❌ invisible to JSON
    ID    string  // ✅ visible to JSON
}
