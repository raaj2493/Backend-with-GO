🎯 The HTTP Protocol: Foundation ⚙️

The primary medium browsers use to talk to servers (send/receive data). HTTP/HTTPS are treated as interchangeable — HTTPS is just a more secure version (TLS encryption, certificates), but underlying principles are the same.

A
Statelessness

Definition: No memory of past interactions. Each request is independent.

Implication: Each HTTP request must carry all necessary information (headers, URLs, methods, auth tokens, cookies, session info) for the server to process it.

Example: To access a user profile, credentials (cookies/tokens) must be sent on every request.

✅ Benefits: Simplicity (server architecture simpler) + Scalability (easy to distribute requests, no single server needs to track a session, if a server crashes, no client state is lost)

Reality Check: Because HTTP is stateless, developers implement state management techniques (cookies, sessions, tokens) for things like logins or shopping carts.

B
Client-Server Model

Client: Browser or app. Initiates communication by sending a request. Provides all needed info (URL, headers).

Server: Hosts resources (websites, APIs). Waits for requests, processes them, sends back responses (web page, data, error, JSON, text, etc.).

⚠️ Rule: Communication is always initiated by the client.

🚂 Underlying Transport & HTTP Versions

TCP: HTTP uses TCP because it's reliable (doesn't lose messages). TCP is connection-based (uses a 3-way handshake).

OSI Model: Backend engineers primarily work at Layer 7 (Application Layer). Network engineering concepts (TCP handshake, TLS) are lower layers (good to know but a rabbit hole).

HTTP 1.0: Each request opened a new connection → inefficient ❌
HTTP 1.1: Persistent connections (reuse same TCP connection) + chunk transfer encoding, better caching ✅
HTTP 2.0: Multiplexing (multiple requests/responses over single connection), binary framing, header compression (HPACK), server push 🚀
HTTP 3.0: Built on QUIC (over UDP, not TCP). Faster connection, less latency, better packet loss handling, no Head-of-Line blocking ⚡

📨 HTTP Messages: Request & Response Structure

Request Components

  • Request Method (e.g., GET, POST)
  • Resource URL (e.g., /users/profile)
  • HTTP Version (e.g., HTTP/1.1)
  • Headers (e.g., Host, User-Agent)
  • Blank Line (signifies end of headers)
  • Request Body (optional data)
GET /users/profile HTTP/1.1
Host: api.example.com
Authorization: Bearer token123

{"userId": 42}

Response Components

  • HTTP Version
  • Status Code (e.g., 200)
  • Status Message (e.g., OK)
  • Response Headers
  • Blank Line
  • Response Body
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 128

{"name": "Alex", "id": 7}
📮 Analogy: Headers = address/phone number on the outside of a parcel. Carriers need metadata without opening the package.

CORS & The OPTIONS Method

Same-Origin Policy: Browsers restrict web pages from making requests to a different domain (origin) than the one serving the page.

CORS: A security mechanism that allows servers to specify who can access their resources and how.

The OPTIONS Method: Used in pre-flight requests to ask the server about its capabilities before sending the actual request. Developers rarely use it directly, but see it in network tabs.

✅ Simple Request Flow

Conditions (all must be true): Method is GET, POST, or HEAD; only simple headers; Content-Type is simple (form, text/plain).

Flow: Browser sends request with Origin → Server checks → responds with Access-Control-Allow-Origin → Browser allows.

⚠️ Pre-flighted Request Flow

Conditions (any true): Method is PUT/DELETE, includes custom headers (Authorization), Content-Type is application/json.

Step 1: OPTIONS preflight with headers: Origin, Access-Control-Request-Method, Access-Control-Request-Headers
Step 2: Server responds with 204 + Access-Control-Allow-Origin/Methods/Headers + Access-Control-Max-Age
Step 3: Actual request (PUT with JSON body)

🔁 Pre-flight caching: Access-Control-Max-Age: 86400 → caches results for 24 hours, saving bandwidth!

💾 HTTP Caching

Definition: Storing copies of responses for reuse to reduce repeated requests, improve load time, reduce bandwidth & server load.

Key Headers:
  • Cache-Control: max-age=10
  • ETag: "abc123" (hash of content)
  • Last-Modified: date
How it works:
1️⃣ Initial request: 200 + ETag + Cache-Control
2️⃣ After expiry: request with If-None-Match: "abc123"
3️⃣ If unchanged → 304 Not Modified (no body)
4️⃣ If changed → 200 with new data + new ETag
📌 Note: Modern client-side solutions (React Query, SWR) give more control, but HTTP caching is still invaluable for simple cases and CDNs.

🗜️ Content Negotiation & Compression

Mechanism for client and server to agree on the best format to exchange data.

Media Type: Accept: application/json → server responds with JSON
Language: Accept-Language: en → server sends English version
Encoding (Compression): Accept-Encoding: gzip, deflate, br → server compresses response, adds Content-Encoding: gzip
📊 Real impact: Without compression: 11,000 entries → 26 MB. With gzip: same file → 3.8 MB (85% savings!) 
GET /large-data.json HTTP/1.1
Host: api.example.com
Accept-Encoding: gzip, br

🔌 Persistent Connections (Keep-Alive)

Problem in HTTP 1.0: Each request required a new TCP connection (slow, resource-intensive).
Solution in HTTP 1.1: Persistent Connections (default). A single TCP connection is reused for multiple requests/responses.

Connection: keep-alive Header: Explicitly asks to keep connection open (though default in HTTP 1.1). Can include timeout and max options.

Connection: close Header: Explicitly closes connection after response (default in HTTP 1.0, still possible in 1.1).

📡 One TCP pipe → many HTTP requests → faster page loads ⚡

📦 Handling Large Requests & Responses

A. Sending Large Data (Client → Server): Multipart Requests

Use case: Uploading files (images, videos, audio). Binary data is transferred in parts.

Content-Type: multipart/form-data; boundary=----WebKitFormBoundaryxyz

Boundary: A delimiter string that separates the different parts of the data in the request body. Indicates where each part starts and ends.

B. Receiving Large Data (Server → Client): Streaming

Use case: Downloading large files, real-time data. Data is sent in chunks over a single persistent connection.

Content-Type: text/event-stream
Connection: keep-alive

Result: Client receives and appends chunks progressively until the complete file is constructed.

🔒 SSL, TLS, HTTPS (Brief Overview)

SSL (Secure Sockets Layer): Original encryption protocol. Now deprecated (security vulnerabilities).

TLS (Transport Layer Security): Modern, secure replacement for SSL. Encrypts data in transit, uses certificates to authenticate server, prevents eavesdropping and tampering. Current recommended version: TLS 1.3.

HTTPS: HTTP + TLS (originally SSL). When you visit an https:// website, TLS encrypts communication between browser and server.

🔐 Padlock icon in browser = TLS-encrypted session
