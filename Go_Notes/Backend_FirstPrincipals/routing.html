🎯 What is routing? (The big picture) 🧩

HTTP methods express the "WHAT" — your intention: do you want to GET data, POST new data, PUT/PATCH update, or DELETE?
Routes (URL paths) express the "WHERE" — the specific resource you're targeting: /users, /products/42, /orders/shipped.
On the server, (METHOD + PATH) together form a unique key that maps to a specific handler function. Same path + different method = different handler. They never collide.

🔁 Example that sticks: GET /books → “please give me all books” vs POST /books → “please create a new book using the data I send”. Same shelf, different action. The router knows the difference because it looks at both the verb and the noun.

📖 Core definition (tape it to your monitor)

Routing = the process of mapping incoming HTTP requests (method + URL) to the appropriate server-side logic. It's the postal service of your API: it reads the address and delivers the request to the right function.

💡 In code: app.get('/users', getUsersHandler) — that’s a route definition.

🔍 The method‑route dance (WHAT + WHERE)

Let's break it down with an actual backend example (Node.js + Express). The server sees every request as a pair:

const express = require('express');
const app = express();

// GET /api/books → fetch all books
app.get('/api/books', (req, res) => {
  res.json([{ id: 1, title: "1984" }, { id: 2, title: "Brave New World" }]);
});

// POST /api/books → create a new book (same path, different method)
app.post('/api/books', (req, res) => {
  // assuming body has { title, author }
  const newBook = req.body;
  // save to database...
  res.status(201).json({ message: "Book created!", book: newBook });
});
🧪 Live mental test: A frontend React app calls fetch('/api/books') → GET runs. It calls fetch('/api/books', {method: 'POST', body: JSON.stringify(book)}) → POST runs. The server uses the method to decide which handler to invoke. That’s why the same URL can do completely different things.

🏷️ Static routes — the constant addresses

Static routes have no variable parts. The URL string is fixed, and the response shape is predictable. Examples: GET /about, GET /api/health, POST /api/auth/login.

Why “static”? Because you always use the exact same string to reach that endpoint. Nothing changes in the path. Great for listing resources or fixed actions.

🌀 Dynamic routes (path parameters)

When you need to identify a specific resource (like a user with ID 42), you use path parameters. Industry‑wide convention: a colon : marks a dynamic segment. Example: /api/users/:userId.

app.get('/api/users/:userId', (req, res) => {
  const userId = req.params.userId; // always a string, even if it looks like a number
  // fetch user from database using userId
  res.json({ id: userId, name: "John Doe" });
});
🌐 Request: GET /api/users/789
→
📦 req.params.userId = "789"
🌍 RESTful readability: GET /users/123 reads as “get me the user with identifier 123”. This semantic clarity is why the colon syntax is universal (Express, FastAPI, Spring, Go Gin, Actix‑web all use it).
📌 Important: Path parameters are part of the URL structure. They are not optional – if the route expects :userId, the request must include that segment. For optional or extra data, we use query parameters.

❓ Query parameters: the Swiss army knife of GET

Since GET requests have no body, we attach extra data as query parameters after ?. Common use cases: pagination, filtering, sorting, search terms.

GET /api/books?page=2&limit=15&sort=title&order=asc&author=Orwell

On the server: req.query.page → "2", req.query.limit → "15".

📚 Pagination metadata (industry standard):
When you request GET /api/books?page=2&limit=20, a well‑behaved API returns:
{ "data": [ ...20 books ], "total": 100, "currentPage": 2, "totalPages": 5, "limit": 20 }
The client uses totalPages to render “Next” / “Previous” buttons. This prevents loading thousands of items at once.
🔍 Why not path parameters for everything? Because path parameters have semantic meaning (user ID, product ID). Query parameters are for modifiers like page, limit, sort field — they don't identify a resource; they modify how the resource is returned.

📂 Nested routes — expressing relationships 🌳

When your resources have parent‑child relationships, you nest them. This makes the API self‑documenting. Example: a blogging platform where users have posts, and posts have comments.

GET /api/users/123
⬇️ “user with ID 123”
GET /api/users/123/posts
⬇️ “all posts written by user 123”
GET /api/users/123/posts/456
⬇️ “single post 456 that belongs to user 123”
GET /api/users/123/posts/456/comments

🎯 Semantic meaning at every level:
• /users → collection
• /users/123 → single user
• /users/123/posts → user's posts collection
• /users/123/posts/456 → a specific post by that user
This nested design scales beautifully and is intuitive for frontend developers.

💡 In Express, you can use express.Router() to nest routes cleanly: app.use('/users', userRouter) and inside that router, define router.get('/:userId/posts', ...).

⚠️ The pain of no versioning

  • You change a field name from name to title
  • Old mobile apps (not updated) crash because they expect name
  • No way to support both old and new clients simultaneously
  • Forced to maintain multiple parallel endpoints with weird names (/getNewProducts, /getOldProducts)

✅ Versioning to the rescue

  • GET /api/v1/products → returns { id, name, price }
  • GET /api/v2/products → returns { id, title, price }
  • You announce: “v1 will be deprecated in 9 months”
  • Frontend teams have a migration window to switch to v2
  • After sunset, v1 returns 410 Gone or redirects to v2
⏳ Real‑world deprecation workflow: Include a Deprecation header in v1 responses: Deprecation: true and a Sunset header with a date. Clients can then plan upgrades without breaking production.
*️⃣

🕸️ Catch‑all routes (the 404 safety net)

What happens when a request hits a route that doesn’t exist? Without a catch‑all, many servers simply return a blank page or a generic HTML error. With a catch‑all, you return a user‑friendly JSON message.

// Place this AFTER all your other route definitions
app.use('*', (req, res) => {
  res.status(404).json({ 
    error: "📭 Route not found",
    message: `Cannot ${req.method} ${req.originalUrl}`,
    tip: "Check your URL or HTTP method"
  });
});
🧩 Why it matters: Instead of a confusing Cannot GET /undefined or a blank page, frontend developers get a clear, actionable error. It also prevents information leakage (stack traces) in production.
🔧 Advanced: You can also add a catch‑all for specific HTTP methods, e.g., app.all('/api/*', ...) to handle any method on any sub‑route.

🚦 Good routing vs messy routing

😖 Anti‑patterns

  • GET /getUser?id=5 (verb in URL)
  • POST /createNewUser (redundant)
  • /api/v1/getAllProducts (no HTTP method usage)
  • No versioning → breaking changes

😎 RESTful best practices

  • GET /users/5 (clean, method implies action)
  • POST /users (create, no extra words)
  • GET /v1/products?page=2 (version + query)
  • Consistent naming: nouns for resources, verbs for methods
✨ Remember: your route is a contract. Make it readable, versioned, and catch‑all friendly. ✨
