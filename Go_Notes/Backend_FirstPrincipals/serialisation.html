🤔 The core problem: communication between different systems 💬

The setup: We have a typical client-server architecture.

  • Client (Frontend): A browser like Chrome, running a JavaScript app (React, Angular, Vue, or any JavaScript framework/library). Most highly interactive client-side apps are built with JavaScript.
  • Server (Backend): Could be running locally or remotely in the cloud (AWS, GCP, Azure). The web server could be built with an entirely different language, for example Rust.

How communication happens: The client communicates with the server through some means of network communication:

  • HTTP (traditional REST API endpoints)
  • gRPC
  • WebSocket

For this series, the focus is on HTTP and traditional REST API-style communication.

📨 The HTTP request structure

A client makes an HTTP request that contains:

• GET or POST method
• The URL (some-domain.com/something)
• If POST: headers + request body

The frontend sends the request → server responds → client parses the response.
But how do they understand each other?

1

🔑 The fundamental question

The key thing to focus on: How can clients and servers make sense of the data being received or sent?

Example scenario:

  • The client is a JavaScript app (dynamic language, not compiled)
  • The server is a Rust app (very strict with types, compiled language)
  • JavaScript and Rust have completely different data types

The challenge: How does data transmitted over a network reach another machine somewhere on the internet, and how is that machine able to make sense of that data, perform business logic, and send a response? And how is the client again able to make sense of that response to show UI or perform its own business logic? How does this all come together?

🌐 The OSI model context (just enough)

Before talking about data transmission over the internet, it's helpful to have some knowledge about the OSI model. If you're a CS student, you might have already encountered it. If you're not, you don't have to completely master it — a high-level overview is enough.

Application Layer (top) ↓ Transport Layer ↓ Network Layer ↓ Physical Layer (bottom)

The same layer structure exists on both client and server. Going too deep is out of scope, but understanding that data travels through layers helps you see why serialization is needed.

2

⚙️ The solution: a common standard

The problem to solve: Two machines in different locations, connected over the internet/cloud. You need to come up with a way so that one machine can send data to another, and they should be able to parse that data and make sense of it in a language-agnostic way.

The obvious and common solution:

Figure out a common standard – a set of rules that both the client and server agree to follow. This is the format that:

  • The client should send data in
  • The server should send responses in
Client side (sending):

JavaScript object → JSON.stringify() → JSON string

Server side (receiving):

JSON string → serde_json::from_str() → Rust struct

The solution identified: Coming up with a serialization standard – a common standard that both client and server agree to for sending and receiving data.

📖 Definition: Serialization & Deserialization

Serialization and Deserialization is basically converting data to and from a common format so that during transmission (or storage):

  • The data is domain-agnostic (or language-agnostic).
  • Machines, servers, and clients from different domains, different languages, and different environments are able to make sense of that data and respond to it.
Serialization: Object → JSON/XML/Protobuf
Deserialization: JSON/XML/Protobuf → Object

🔤 Understanding JSON (JavaScript Object Notation)

JSON stands for JavaScript Object Notation. Despite its name, it's used everywhere — not limited to JavaScript. Common use cases:

  • Configuration files
  • HTTP REST API transmission between clients and servers
  • Log files (very popular choice for logging application data)

📐 JSON structure rules (must remember):

  • Starts with an opening curly brace { and ends with }
  • All keys must be inside double quotes and must be strings (you cannot make any other data type a key)
  • Values can be: string (double quotes), number, boolean, array, or another object (nested JSON)
  • Example of nested JSON: { "address": { "country": "India", "phoneNumber": 3456 } }
3

🧪 Practical demonstration — JSON in action

Using a backend suite interface, let's see a POST /api/books request/response flow.

// Request body (JSON sent by client)
{
  "id": 4,
  "title": "some title",
  "author": "some author"
}

Observations: opening/closing braces, all keys double‑quoted, values: number and strings.

🧠 Mental model for backend engineers (regarding OSI model):

If you went deep into the OSI model, you'd know: from the application layer, data gets converted into data frames → IP packets → bits → voltage signals over optical fiber. But as a backend engineer, your responsibility ends at the application layer where data gets converted into JSON. The server only ever reads the JSON — you don't have to worry about the intermediary steps (0s and 1s, frames, etc.).

// Response body (JSON sent by server)
{
  "books": [
    { "id": 1, "title": "1984", "author": "Orwell" },
    { "id": 4, "title": "some title", "author": "some author" }
  ]
}

The server understands the request JSON, processes it, and responds with another JSON. The client receives this data, parses it (e.g., JSON.parse() in JavaScript), and renders the UI. This is how JSON flow works in real‑life client‑server models.

📌 Final summary — what you must remember

What is Serialization and Deserialization?
Serialization and deserialization are the techniques using which data is converted into a common format/standard format so that:

  • Clients can send data to the server in that format
  • Servers can receive that data, make sense of it, parse it, and perform business logic
  • Servers can send data back in that same common format
  • Clients can again do the same – receive, parse, and make sense of it

🎯 The core idea:

Serialization and deserialization deal with a standard format so that data is understandable across domains and languages.

Key takeaway: Serialization and deserialization don't have a lot of concepts. It is a phenomenon in the backend world that exists – you have to know that it exists. The entire flow of:

  • JSON being transmitted by the client to the server
  • The server parsing it, understanding it, making sense of it
  • The server responding with appropriate data to the client
  • The client again parsing it, understanding it, and making sense of it

This whole flow is called serialization and deserialization. That's all there is to it. ✨
